> mysql 8+ 删除了查缓存

1. 连接器：处理来自客户端的连接，检查权限；
2. 查缓存：最近是否执行过这个sql语句，命中则直接返回结果；
3. 缓存不命中，分析器检查slq语法，关键字词法分析；
4. 优化器：进行索引选择，确定多个表的连接顺序, 生成执行计划；
5. 执行器：调用存储引擎接口，拿数据返回给客户端。

## 索引
### 1. 索引有哪些种类？
[[database索引分类]]
索引从三个维度分类, 一个索引的三个属性
1. 索引的底层数据结构实现：B+树索引，全文索引；
2. 叶子节点的物理存储：聚集索引、非聚集索引；
3. 索引的逻辑功能：普通索引、唯一索引、主键索引、联合索引。
### 2. MySQL使用B+树来作索引，优势是什么？
设计索引目的：减少数据库磁盘IO次数
1. 磁盘IO次数少：非叶子节点只存储索引key，不存数据。单个节点可容纳更多key，树高度更低，磁盘IO次数少。
2. 范围查询性能高：数据存储在叶子节点，叶子节点通过链表连接，通过索引找到起始节点后可以顺序遍历，无需回溯父节点。同时根到叶子节点得路径长度相同，查询性能稳定。
3. B+树得调整成本低：插入删除操作发生在叶子节点，结构调整范围小，维护成本低于B树。
```markdown
B树
        [30]
       /    \
 [10 20]   [50 70]
           /  |  \
        40 60 80
- 非叶子节点保存key和data，单个节点可容纳key少，树的高度更高
- 范围查询性能差,遍历得时候要回溯父节点，对树进行中序遍历
- 查询性能不稳定, 有可能在根节点就找到，也可能要找到底；
- 可能对枝干节点进行插删操作，会对B树结构进行大调整
B+树 
            [30 50]
         /   |   \
     ...   ...   ...

叶子层：

10 -> 20 -> 30 -> 40 -> 50 -> 60 -> 70 -> 80
```
### 3. 什么时候需要创建索引，什么时候不需要创建索引？
索引的目的：提高查询性能。(不要扫全表，通过索引来查询)
但有副作用：
1. 降低插入、修改、删除的性能；
2. 需要存储索引

因此，查询多，写入少的表需要建立索引。如
1. 数据量大，数据区分度高的表，建立索引。
2. 写入频繁的表就不适合建立索引。


### 4. 索引失效的场景有哪些？
B+Tree索引查找本质是 \[对有序的索引\] 进行二分查找。
索引失效：
1. 破坏了破坏B+Tree索引的有序性，无法进行二分查找；
	如：
	1. 对索引列做运算/函数/类型转换。
	2. like最左模糊匹配；
2. 优化器估算索引查询成本比扫全表更高，放弃索引查询。
	如：
	1. 使用or条件跨列查询，需要合并多个索引的结果集，成本高。
	2. 数据量小、区分度低的表，还不如扫全表。

## 事务
### 1. 事务的四大特性有哪些？
AIDC
- **原子性：**
	**一个事务的所有操作要么全部完成，要么全部没完成。** 只要有一个操作失败了就会回到原点，不存在中间态。
- 隔离性：
	多个事务并发执行，互不干扰。数据不会因此错乱。
- 持久性：
	事务提交后，对数据的修改是永久性的。数据库故障也不会因此丢失更改。
- 一致性：
	事务执行前后数据库的状态保持一致。转账前张三和李四的总钱数是 1000 元，转账后不管成功还是失败，两人的总钱数**必须还是 1000 元**。
### 2. 说说事务隔离级别
事务隔离是为了解决：多事务并发执行产生数据错乱的问题。
数据错乱问题有
1. 脏读：读到了未提交的数据。
2. 不可重复读取：同一条记录，多次查询不一致。
3. 幻读：同一范围的记录，多次查询不一致。

隔离级别：
1. 读未提交：可以读取未提交的更改 => 三种错乱都有；
2. 读提交: 只能读取提交后的更改 => 解决脏读
3. 可重复读: 使事务执行过程中读取的数据，与事务启动时的数据一致。=> 解决脏读，不可重复读；
4. 串行化：给读写操作加锁。数据冲突时，需要等待事务完成后才会继续执行。

## 锁与MVCC
### 1. MySQL有哪些锁，作用是什么？
为什么需要锁：多事务并发执行，数据会错乱。

数据库锁
- 全局锁：对整个数据库加锁，数据库只读，用于备份数据库，保持数据备份前后一致性。
表锁
- 元数据锁：保护表结构；
- 意向锁：这个表已经加了行锁的标识，这个表里面有人持有行锁。当事务要加表锁时会等待，无法继续加表锁；
- AUTO-INC 锁：对递增字段加锁，保证多事务执行时这个字段仍是递增状态；
行锁
- 记录锁：用于一行记录的加锁；
- 间隙锁：锁范围查询的区间，防止区间插入数据, 出现幻读；
- Next-Key Lock：锁范围查询的区间及区间的右边界`(...]`，防止区间插入与右边界修改，出现幻读；
- 插入意向锁：提高多事务并发插入的效率。在被锁的区间做插入意向标识，一旦解锁可快速插入。
### 2. 说说MVCC多版本并发控制? ⭐
通过加锁来解决多事务并发执行，在高并发时性能很低。MVCC解决这个问题。
MVCC机制：事务开始时对数据进行“快照”，每个事务都有各自的快照，互不干扰。因此无需等待，实现了多事务并发执行。

 **原理：**
> 历史版本链+ReadView

修改后的旧数据会放入回滚日志undoLog。
每一版的数据都会存储
1. 事务`trx_id`
2. 回滚指针`roll_pointer` : 指向上个版本的UndoLog

事务开始时，进行快照产生ReadView
ReadView中存储了
	`creator_trx_id`: 当前事务id
	`m_ids` : 所有活跃事务的id
	`min_trx_id` 最久远的活跃事务id
	`max_trx_id` 未来下一个事务的id

通过`ReadView`来确定当前事务可以查看哪一版本的数据
1. 这版是自己修改的吗? `这版的trx_id == creator_trx ?` => yes, 可以；
2. 这版是readview创建前(事务开始前)就提交了吗？`trx_id < min_trx ?` =>yes,可以
3. 这版是readview创建后(当前事务开始后)才存在的吗？`trx_id >= max_trx ?` => yes, 不可读取；
4. 这版是已经提交了吗？这版id在`[min_trx_id, max_trx_id]` && `m_ids`, 则这版数据修改中未提交 => 不可读取。
## MySQL
### 1. MySQL执行引擎有哪些？
MySQL常见存储引擎有InnoDB、MyiSAM和Memory。
- InnoDB是MySQL默认存储引擎，支持事务AIDC、行锁和外键；
- MyISAM不支持事务和行锁，只支持表锁；
- Memory将数据放在内存中，数据处理速度很快，但服务重启后数据会丢失。

### 2. MySQL日志文件有哪些？
**存储引擎层（InnoDB）**
1. undo log回滚日志: `Innodb` 存储引擎层生成的日志。用于事务的原子性事务回滚，用户多版本并发控制MVCC的版本链。
2. redo log重做日志: 物理日志，记录了某个数据页做了什么修改。用于事务的持久性崩溃恢复，数据库崩溃重启后会读取redo log将未写入磁盘的数据redo, 保证数据不丢失。

**Server层**
1. binlog归档日志：Server 层生成的日志, binlog记录所有的 DDL（建表等）和 DML（增删改）。 用来进行数据恢复/备份。
2. relay log中继日志：`slave`从库通过io线程拷贝master主库的 `binlog` 后本地生成的日志。用于主从复制场景下。
